## মন্টু মিয়াঁর সমস্যা

মন্টু মিয়াঁ তার লোড ব্যালেন্সার সেটআপ করার জন্য রেডি। কিন্তু তার মনে খটকা — "এত টাকা খরচ করে এই জিনিস বসাচ্ছি, এ কি শুধু ট্রাফিক পুলিশগিরিই করবে? নাকি আরও কোনো গুণ আছে?"

একটু ঘাঁটাঘাঁটি করতেই মন্টু দেখল, লোড ব্যালেন্সার আসলে 'Swiss Army Knife'-এর মতো। এর কাজের কোনো শেষ নেই। মন্টু তার নোটখাতায় লোড ব্যালেন্সারের জাদুকরী সব ক্ষমতার কথা টুকে নিল।

## ট্রাফিক বাড়লে সার্ভার বাড়ে (Auto Scaling)

মন্টু ভাবল, "সামনেই বিড়ালটিউবের সুপারস্টার 'মিস্টার ট্রিট'-এর জন্মদিন। ও যদি ঘোষণা দেয় যে রাত ১২টায় লাইভে আসবে, তখন কী হবে?" নিশ্চিতভাবেই দুনিয়ার সব বিড়ালপ্রেমী একসাথে হুমড়ি খেয়ে পড়বে। মন্টুর ১০টা সার্ভার তখন নিমেষেই ক্র্যাশ করবে।

এখানেই লোড ব্যালেন্সারের ম্যাজিক। মন্টু লোড ব্যালেন্সারকে বলে রাখতে পারে, "দেখো বাপু, যখনই দেখবে সার্ভারগুলোর ওপর চাপ (CPU Usage) ৮০% এর বেশি হয়ে গেছে, সাথে সাথে আরও ১০টা নতুন সার্ভার চালু করে দেবে।" আবার লাইভ শেষ হয়ে গেলে যখন চাপ কমবে, তখন অতিরিক্ত সার্ভারগুলো বন্ধ করে দেবে। এই পুরো ব্যাপারটা অটোমেটিক ঘটবে, মন্টুকে রাতে ঘুম থেকে উঠে সার্ভার অন-অফ করতে হবে না।

```
ট্রাফিক কম  → সার্ভার ২টা running
ট্রাফিক বাড়ছে → auto scale → সার্ভার ১০টা
ট্রাফিক কমল  → auto scale down → সার্ভার ২টায় ফিরে এলো
```

## সার্ভার মরলেও সাইট বাঁচে (Health Checks)

মন্টুর বন্ধু রনি একদিন জিজ্ঞেস করল, "ধরো তোমার একটা সার্ভার হঠাৎ ক্র্যাশ করল। তখন কী হবে? পুরো সাইট ডাউন?"

মন্টু মাথা নাড়ল, "না না, লোড ব্যালেন্সার সবসময় প্রতিটা সার্ভারকে চেক করতে থাকে। প্রতি ৩০ সেকেন্ডে একবার জিজ্ঞেস করে, 'এই সার্ভার, তুমি কি ঠিকঠাক আছ?'"

সার্ভার যদি জবাব না দেয় বা 'আমি অসুস্থ' বলে, লোড ব্যালেন্সার সাথে সাথে সেই সার্ভারে নতুন রিকোয়েস্ট পাঠানো বন্ধ করে দেয়। বাকি সুস্থ সার্ভারগুলো কাজ চালিয়ে যায়। ব্যবহারকারীরা কিছুই টের পায় না।

## অ্যালগরিদম: কীভাবে ঠিক করে কাকে কাজ দেবে?

লোড ব্যালেন্সার কীভাবে বুঝবে কোন সার্ভারকে কোন রিকোয়েস্ট দেবে? এর জন্য বিভিন্ন অ্যালগরিদম আছে।

**Round Robin** — পালা করে সবাইকে দাও:

```
Request 1 → Server A
Request 2 → Server B
Request 3 → Server C
Request 4 → Server A (আবার শুরু থেকে)
```

**Least Connections** — যে সার্ভার এখন সবচেয়ে কম ব্যস্ত, তাকে দাও।

**IP Hash** — একই ব্যবহারকারীকে সবসময় একই সার্ভারে পাঠাও (session consistency-র জন্য দরকারি)।

## SSL Termination — এনক্রিপশনের ঝামেলা কমায়

HTTPS মানে প্রতিটা রিকোয়েস্ট এনক্রিপ্ট-ডিক্রিপ্ট করতে হবে। এটা CPU-intensive কাজ। প্রতিটা সার্ভারে এই কাজ করলে অনেক সম্পদ নষ্ট হয়।

লোড ব্যালেন্সার এখানে বলে, "তোমরা চিন্তা করো না, SSL-এর ঝামেলা আমি নিজে সামলাব।"

```
ব্যবহারকারী ──HTTPS──► Load Balancer ──HTTP──► সার্ভারগুলো
```

লোড ব্যালেন্সার HTTPS ডিক্রিপ্ট করে HTTP হিসেবে সার্ভারে পাঠায়। সার্ভারগুলো নিরাপদ internal নেটওয়ার্কে থাকে, তাই HTTP-ই যথেষ্ট।

## মন্টুর উপলব্ধি

নোটখাতা বন্ধ করে মন্টু বলল, "লোড ব্যালেন্সার শুধু ট্রাফিক পুলিশ না, এটা একটা পুরো সিকিউরিটি গার্ড, হেলথ মনিটর, এবং অটোমেশন ইন্জিনিয়ার — একসাথে।"

যে কোনো প্রোডাকশন অ্যাপ্লিকেশন স্কেল করতে চাইলে লোড ব্যালেন্সার ছাড়া কোনো উপায় নেই। এটা বিলাসিতা না, এটা প্রয়োজন।
