## রহিম সাহেবের লাইব্রেরি

রহিম সাহেব একটি বিশাল লাইব্রেরির মালিক। লাইব্রেরিতে লক্ষাধিক বই আছে, কিন্তু কোনো সূচি নেই। কেউ যদি "ডেটা স্ট্রাকচার" বিষয়ক কোনো বই খুঁজতে আসে, রহিম সাহেবকে প্রথম তাক থেকে শুরু করে শেষ তাক পর্যন্ত প্রতিটা বই দেখতে হবে।

এটাই হলো ইন্ডেক্সবিহীন ডেটাবেজ কোয়েরির অবস্থা।

---

## Full Table Scan — সব বই একে একে খোঁজা

ধরো তোমার `users` টেবিলে ১ কোটি রেকর্ড। তুমি লিখলে:

```sql
SELECT * FROM users WHERE email = 'nokib@example.com';
```

ইন্ডেক্স না থাকলে ডেটাবেজ প্রথম রো থেকে শেষ রো পর্যন্ত প্রতিটা চেক করবে — মানে ১ কোটি comparisons। এটাই **Full Table Scan**। ডেটাবেজের ভাষায়: O(n)।

রহিম সাহেব যেটা করে তার নতুন সহকারীকে — "এই ছেলে, তুমি এক নম্বর তাক থেকে শুরু করো, আর আমি শেষেরটা থেকে।" দুজন মিলেও কাজটা ধীর।

---

## ইন্ডেক্স কী — সূচিপত্র বানানো

রহিম সাহেব একদিন বুদ্ধি খাটালেন। তিনি একটা আলাদা খাতা বানালেন — **বিষয়ের নাম → তাক নম্বর**। এখন কেউ "JavaScript" খুঁজলে সরাসরি সূচিতে দেখে নো. 47 তাকে যায়। লাখো বই হলেও সময় লাগে মাত্র সেকেন্ড।

ডেটাবেজে ইন্ডেক্স ঠিক এইভাবে কাজ করে:

```sql
-- email কলামে ইন্ডেক্স তৈরি করো
CREATE INDEX idx_users_email ON users(email);

-- এখন এই কোয়েরি O(n) থেকে O(log n) হয়ে গেল
SELECT * FROM users WHERE email = 'nokib@example.com';
```

ডেটাবেজ একটি **B-Tree** (Balanced Tree) ডেটা স্ট্রাকচার ব্যবহার করে। এটা বাইনারি সার্চের মতো — প্রতিটা ধাপে অর্ধেক ডেটা বাদ দেয়।

---

## কোথায় ইন্ডেক্স লাগাবে?

**✅ লাগাও:**
- `WHERE` clause-এ যে কলামগুলো বারবার আসে
- `JOIN` করার কলামগুলোতে (foreign keys)
- `ORDER BY` এবং `GROUP BY`-তে যা বেশি ব্যবহার হয়
- Unique constraint আছে এমন কলামে (email, username)

**❌ লাগাবে না:**
- যে কলামের ডেটা অল্প unique (যেমন: gender — মাত্র ২-৩ ভ্যালু)
- প্রচুর Write হয় এমন টেবিলে অতিরিক্ত ইন্ডেক্স
- ছোট টেবিলে (হাজারের কম রেকর্ড — Full Scan-ই দ্রুত)

---

## ইন্ডেক্সের ডার্ক সাইড

রহিম সাহেবের সূচিখাতা রাখার একটা ঝামেলা আছে — নতুন বই এলে বা পুরোনো বই বাদ দিলে, সূচিটাও আপডেট করতে হয়। এটা সময় নেয়।

ডেটাবেজেও তাই। প্রতিটা `INSERT`, `UPDATE`, `DELETE` অপারেশনে ইন্ডেক্সও আপডেট হয়। সুতরাং:

- **Read-heavy** অ্যাপ → বেশি ইন্ডেক্স = দ্রুত
- **Write-heavy** অ্যাপ → বেশি ইন্ডেক্স = ধীর লেখা

ব্যালেন্স করতে হবে।

---

## Composite Index — একসাথে একাধিক কলাম

রহিম সাহেবের সূচিতে যদি "বিষয় + লেখকের নাম" দুটো মিলিয়ে খোঁজার ব্যবস্থা থাকে:

```sql
-- দুটো কলাম মিলিয়ে ইন্ডেক্স
CREATE INDEX idx_posts_user_date ON posts(user_id, created_at);

-- এই কোয়েরি ইন্ডেক্স ব্যবহার করবে ✅
SELECT * FROM posts WHERE user_id = 5 ORDER BY created_at DESC;

-- এই কোয়েরি ইন্ডেক্স ব্যবহার করবে না ❌ (user_id ছাড়া)
SELECT * FROM posts WHERE created_at > '2026-01-01';
```

**Composite index সবসময় বাম দিক থেকে শুরু হতে হবে।**

## EXPLAIN দিয়ে যাচাই করো

PostgreSQL বা MySQL-এ `EXPLAIN` কমান্ড দিয়ে দেখতে পারবে ডেটাবেজ আসলে কী করছে:

```sql
EXPLAIN SELECT * FROM users WHERE email = 'nokib@example.com';
```

যদি আউটপুটে `Seq Scan` দেখো — ইন্ডেক্স ব্যবহার হচ্ছে না।
যদি `Index Scan` দেখো — ইন্ডেক্স কাজ করছে। ✅

## উপসংহার

রহিম সাহেব এখন তার লাইব্রেরির প্রতিটা বিষয়ের জন্য আলাদা সূচি রাখেন। খোঁজাখুঁজিতে সময় লাগে না, কিন্তু নতুন বই ঢোকানোর সময় একটু বেশি কাজ হয়।

ডেটাবেজ ইন্ডেক্সও সেই একই trade-off। সঠিক কলামে সঠিক ইন্ডেক্স লাগালে তোমার ধীর কোয়েরি মুহূর্তেই দ্রুত হয়ে যাবে।
